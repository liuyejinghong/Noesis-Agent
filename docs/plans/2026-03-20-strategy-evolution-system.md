# QPorter Lite 策略进化系统 — 可行性探讨

**日期：** 2026-03-20
**状态：** 可行性探讨中，尚未形成执行方案

---

## 背景

用户提供基础策略，系统负责在实盘运行过程中持续收集交易数据，定期分析策略表现，并基于分析结果提出策略改进方案。改进不仅限于参数调整，可能涉及策略骨干逻辑变更（如增减过滤条件、改变入场/出场规则、切换策略类型）。所有变更需要人工审批后才应用。

## 核心设计

### 人的角色：审批者

- AI 负责分析、提案、验证
- 人只在关键节点做审批：策略变更、实盘上线
- 人随时可查看全局状态但不需要主动操作

### 三个子系统

#### 1. 实盘交易分析引擎（统计驱动，不用 LLM）

从交易行为记录中提取：
- 入场信号与当时行情特征
- 持仓期间行情走势
- 出场方式（止盈/止损/超时/信号反转）
- 盈亏金额和比率
- 市场环境标注（趋势/震荡/高波动/低波动）

输出分析报告：
- 按环境分组的胜率和盈亏比
- 持仓时间分布
- 连续亏损分析
- 止损效率评估
- 策略强项与弱项诊断

频率：月级，交易记录累积 50-100 笔后触发

#### 2. 策略改进提案引擎（LLM 驱动）

输入：
- 当前策略源码
- 分析报告
- 历史交易明细

输出：改进提案（Proposal），包含：
- 发现的问题
- 改进方向
- 具体代码变更
- 预期效果

改进范围不限于参数调整，可能包括：
- 增加过滤条件（如 ADX 过滤震荡环境）
- 改变止损方式（如固定百分比改为 ATR 动态止损）
- 增加趋势确认条件
- 根本性策略类型变更（需审批）

#### 3. 验证与审批闭环

提案生成后：
1. 自动生成新策略版本（带版本号）
2. 自动回测，和旧版本对比
3. 对比结果 + 提案推送给人审批（Telegram / Web）
4. 审批通过 → 部署到测试网验证
5. 测试网运行一段时间 → 推送验证报告
6. 再次审批 → 切换实盘策略

### 调参 vs 策略进化的区别

| | 调参 | 策略进化 |
|---|---|---|
| 改什么 | 数值参数 | 代码逻辑 |
| 输入数据 | 历史 K 线 | 实盘交易记录 + 市场环境 |
| 驱动方式 | Optuna 搜索 | LLM 分析 + 提案 |
| 频率 | 随时可跑 | 一两个月一次 |
| 风险 | 低（参数空间有界） | 高（代码变更不可预测） |
| 审批需求 | 可选 | 必须 |

### 心跳机制

两种频率：
- 分钟级心跳：健康检查、自动排错、状态同步
- 月级心跳：触发交易分析、策略进化提案

### AI 驱动 OpenCode 的定位

不用于实时驱动交易系统，而用于策略迭代：
- AI agent 在独立 worktree 里改策略代码、跑回测、生成报告
- 改完后提交提案给人审批
- AI 的不确定性限制在开发环境，不影响生产环境

## 风险

- LLM 改出来的策略可能回测很好但实盘很差（过拟合）
- 交易记录样本量不足时分析结论不可靠
- 全闭环状态机设计复杂度高

## 建议实施路径

### 第一期：被动分析

- 每月自动生成实盘交易分析报告
- 推送到 Telegram / Web
- 人看报告后手动决定是否改策略
- 工作量：Small

### 第二期：主动建议

- LLM 读分析报告 + 当前策略 → 输出改进提案
- 提案自动回测验证
- 验证结果 + 提案推送给人审批
- 人审批后手动应用
- 工作量：Medium

### 第三期：半自动闭环

- Telegram 审批后系统自动应用新策略并部署测试网
- 测试网运行后自动推送验证报告
- 再次审批后自动切换实盘
- 工作量：Medium

## 探讨记录

### 2026-03-20：以 R-Breaker 作为核心策略的可行性

**结论：可行性比从零生成策略显著更强。**

R-Breaker 是公认的经典日内反转+趋势突破混合系统，骨架确定，弱点已知，有公开基准可对比。AI 不需要发明交易逻辑，只需要在已验证的框架上做针对性改进。

AI 可以发挥价值的改进维度：
- 系数动态化（根据波动率调整 6 个价位的计算系数）
- 反转/趋势模式的切换条件智能化
- 时间过滤（某些时段/环境不交易）
- 止损止盈参数按市场环境自适应
- 弱点针对性修复（如震荡日假突破、强趋势日反转亏损）

关键适配问题：R-Breaker 原版是日内策略，加密市场 24h 交易。"前一日"的定义本身是一个可搜索维度：
- UTC 0:00 切割（日线收盘标准）
- UTC+8 切割（操作者时区）
- 按波动率极值时刻动态切割
- 按亚盘/美盘/欧盘切换节奏切割

这些定义不需要人来猜，可以作为 AI 回测搜索的参数之一，让数据给结论。全球化市场的时区博弈本身就是一个 alpha 来源。

### 2026-03-20：多品种多市场扩展

**用户原话：** 不只做一个品种，也不只做加密货币。A 股、港股、美股，只要允许 API 自动交易，都想利用起来。资金闲置也是一种损失。多品种扫描可以更频繁地得到交易信号。

**关键洞察：R-Breaker 天然适合多品种多市场。** 它只需要 OHLC 数据，不依赖品种特性，不需要 Level 2 行情。只要一个市场有日线数据和下单 API，R-Breaker 就能跑。

#### 多品种场景下因子/特征提取的角色变化

单品种时因子不重要（R-Breaker 不靠因子驱动）。多品种时因子变成三件事的基础：
- **品种筛选**：波动率、流动性、趋势强度 → 选出当日值得交易的品种
- **仓位分配**：品种相关性、波动率、信号强度 → 决定资金怎么分
- **组合风控**：总敞口、相关性、回撤 → 多品种同时持仓时的风险管理

#### 各市场 API 可行性（针对中国大陆个人开发者）

| 市场 | 可行性 | 接入方式 | 门槛 | 主要限制 |
|------|--------|---------|------|---------|
| Binance 加密货币 | ✅ 最容易 | ccxt（REST + WebSocket） | 仅需 KYC | 中国大陆 IP 受限，需海外手机号 |
| 美股 (IBKR) | ✅ 可行 | ib_insync / Web API | 开户 + 入金 | 资金跨境受外汇管制（年 5 万美元） |
| 美股 (Alpaca) | ✅ 可行 | REST + WebSocket | 护照即可，$1 起 | 仅美股现货，无期权/期货 |
| 港股 (富途) | ✅ 可行 | futu-api + OpenD 网关 | 富途账户 | 对中国用户最友好 |
| 港股 (长桥) | ✅ 可行 | HTTP + WebSocket + SDK | 长桥账户 | 声称 AI-Friendly（MCP 支持） |
| A 股 (QMT) | ⚠️ 有门槛 | xtquant + MiniQMT | 10-50 万资产 | 必须通过券商系统，私有协议 |
| A 股 (PTrade) | ⚠️ 有门槛 | 恒生云端托管 | 同上 | 策略运行在券商服务器 |

#### 没有统一的跨市场封装

- 加密货币：ccxt 已统一
- A 股：xtquant（私有）
- 港股：futu-api 或 longbridge（各自独立）
- 美股：ib_insync 或 alpaca-trade-api

**现实：每个市场需要独立的适配层。** 工程上可以抽象出统一的 Adapter 接口（数据获取 + 下单 + 查账户），各市场各自实现。

#### 建议的扩展优先级

1. **先在加密货币内做多品种**（BTC/ETH/SOL/...） — 接口已有（ccxt），扩展成本最低
2. **接美股/港股**（IBKR 一个账户覆盖两个市场） — 成熟 API，但需新适配层
3. **A 股最后**  — API 限制最多，合规风险最大

#### AI 在多市场场景下的额外价值

- **跨品种分析**：BTC 和 ETH 同时出信号，历史上同时做的效果好还是只做一个好？
- **跨市场分析**：美股在跌、加密在涨，这个宏观环境下 R-Breaker 在哪个市场更可靠？
- **动态配置**：根据各市场近期表现自动调整资金分配比例
- **统一风控**：所有市场的总敞口、相关性、回撤合并管理

### 2026-03-20：架构原则与模块化要求

**用户核心诉求：** 不想再重构。新市场、新策略都应该是"加一个文件"，不是"改一堆核心代码"。

#### 架构原则

1. **市场适配层插件式** — 统一接口（行情、下单、查账户、查持仓），每个市场独立实现
2. **策略仓库插件式** — R-Breaker 只是第一个，后续还有海龟、Dual Thrust 等，每个策略独立注册
3. **同一策略可有市场特化版本** — R-Breaker 的 BTC 版和 A 股版参数/逻辑可能不同
4. **特征提取用成熟开源库** — 不造轮子
5. **AI 是最上层编排者** — 选策略、选市场、选参数，不负责底层执行
6. **最终愿景** — AI 自动编排多策略 × 多市场 × 自动匹配，根据市场环境自动选最佳组合

#### 整体架构愿景

```
┌─────────────────────────────────────────────────────┐
│                  AI 编排层                           │
│  市场环境 × 策略历史表现 → 自动选策略+市场+参数       │
├─────────────────────────────────────────────────────┤
│  策略仓库（插件式）           市场适配层（插件式）      │
│  ┌──────────┐               ┌──────────┐           │
│  │R-Breaker │               │ Binance  │           │
│  │ ├ BTC版  │               ├──────────┤           │
│  │ ├ ETH版  │               │ IBKR     │           │
│  │ └ A股版  │               ├──────────┤           │
│  ├──────────┤               │ 富途     │           │
│  │ 海龟     │               ├──────────┤           │
│  ├──────────┤               │ QMT      │           │
│  │ Dual Thrust│             └──────────┘           │
│  └──────────┘                                      │
├─────────────────────────────────────────────────────┤
│              特征/环境识别层（开源库）                 │
│  技术指标 + 市场状态分类 + 波动率分析 + 相关性分析     │
├─────────────────────────────────────────────────────┤
│              统一风控层                               │
│  跨市场总敞口 / 相关性 / 回撤 / 资金分配              │

```

### 2026-03-20：开源工具链调研

**核心结论：有成熟方案可用，不需要造轮子。**

#### 推荐技术栈

| 层 | 推荐方案 | Stars | pip |
|---|---|---|---|
| 技术指标 | **TA-Lib** (工业标准) | 11.8k | `pip install TA-Lib` |
| 特征工程 | **tsfresh** (800+ 自动特征) | 5.7k | `pip install tsfresh` |
| 市场状态检测 | **HMM 自建** (hmmlearn) | — | `pip install hmmlearn` |
| 因子分析 | **Alphalens** (因子验证) | 4k | `pip install alphalens-reloaded` |
| 综合框架 | **Qlib** (微软出品) | 39k | `pip install qlib` |
| 向量化回测 | **VectorBT** (Numba 加速) | 6.9k | `pip install vectorbt` |

#### 警告

- **pandas-ta 已废弃** — 原作者失联，PyPI 被接管，存在供应链风险
- **市场状态检测没有生产级开源库** — 需 HMM 或规则引擎自建，工程量不大

#### 这些工具在本系统中的定位

不是用来驱动交易信号（信号由 R-Breaker 等策略产生），而是用来：
1. 给 AI 分析提供结构化输入 — 每笔交易带市场环境标签
2. 做策略的过滤器和增强器 — 环境不适合时暂停交易
3. 多品种场景下做品种筛选和仓位分配
4. 跨市场宏观环境识别

### 2026-03-20：Oracle 框架评估（一针见血版）

**总体判断：零件对，装配错。**

框架把最不可靠的部分（AI 判断力）放在了架构最顶层，而最可靠的部分（规则引擎、风控、执行器）在底下。等于把整个系统的成败押在 AI 做决策上——而 2026 年的 AI 擅长分析和建议，不擅长做交易决策。

#### 逐维度评估

**框架可行性：零件对，装配错。** 插件式策略/市场/风控都是标准做法。但 AI 编排器不应在架构中心，应在边缘（分析、报告、建议）。审批流程缺硬性指标：什么时候必须拒绝？必须回滚？样本量最少多少？没定义就会变成走过场。

**AI 角色：好助手，差决策者。** 能可靠做：总结交易日志、聚类失败案例、起草代码改动、建议参数搜索空间、生成报告。做不好的：从噪音数据因果推断、发现稳健 alpha、跨市场资金分配、判断改进是真的好还是只是看起来好。让 AI 当 CIO，它会自信地过拟合。

**R-Breaker：好起点，不是终点。** 用已验证系统起步是对的。但风险不在于"公开"，而在于可能把"知名"混淆成"扣除手续费和滑点后仍能赚钱"。公开系统可做脚手架，很少能直接当 edge。

**多市场复杂度：被严重低估。** 不同交易时段、日历、保证金模型、币种、合规要求、券商语义，会让"统一接口"变成一堆硬边界情况。当前代码离多市场还差很远。

**数据驱动进化：可提假设，不可做决策。** 50-100 笔实盘交易够做描述性诊断，不够做置信度高的策略结构变更。用这个样本量改代码级逻辑是经典过拟合陷阱——必须再拿到更大历史数据做 walk-forward 验证。

**开源工具链：部分合理，部分过度。** TA-Lib + VectorBT 合理。tsfresh 危险（爆炸式生成特征，假阳性更容易）。Qlib 太重（研究平台，不是轻量规则系统需要的）。Alphalens 只在真正做因子排序时才有价值。

#### 三大灾难性失败模式

1. **跨市场风控撒谎**：某适配器数据延迟或语义错误，你以为敞口在限制内，实际已超
2. **AI 提案过拟合**：回测好、测试网好、一上实盘就亏——因为本质是在小样本上讲了一个好听的故事
3. **市场结构不匹配**：日线切割、交易时段、杠杆假设、K 线缺失等细节层面系统性错误，不是随机亏损，是机械性自我毁灭

#### Oracle 建议的正确路径

1. **v1 删掉"AI 顶层编排"**，改为"AI 离线研究助手"——写报告、提建议、起草代码，不做实时决策
2. **冻结在一个市场一个资产类别**，不建跨市场编排，直到当前 backtest → testnet → live 路径无聊地稳定
3. **审批流程加硬性门控**：必要样本量、稳健性检查、拒绝路径、回滚路径、熔断行为
4. **实盘记录只用来生成假设**，每个提议的改动都要在更大历史数据上做 walk-forward + 费用/滑点假设验证
5. **工具链砍到最小**：保留 TA-Lib 和回测栈，状态标注只有证明增量价值时才加，Qlib/Alphalens/tsfresh 推迟到真正需要时
6. **R-Breaker 当基准而非假定 alpha**：如果它在目标交易所扣费后不赚钱，停止扩展架构，先修研究前提

#### 何时可以扩展

当且仅当：一个市场稳定运行、reconcile 可靠、研究工作流可重复、简单状态过滤器被证明有增量价值时——才可以加第二个密切相关的市场。如果这些条件不满足，加市场或让 AI 改代码是在扩展不确定性，不是扩展能力。

### 2026-03-20：方向调整——基于 Oracle 评估后的决策

#### 市场范围收窄

**决策：只做加密货币。** 不做全球金融市场。

- CEX 优先适配 Binance（已有）
- DEX 优先适配 Hyperliquid（链上永续合约，API 完整）
- 仍然模块化，预留接口，但不急于扩展
- 核心：先在 Binance 单市场跑通跑稳

选加密货币的理由：准入门槛最低、最自由、CEX→DEX 扩展复杂度远低于跨到股票市场（数据格式接近、交易时间一致、结算逻辑相似、合规压力最小）。

#### 项目定位变化

QPorter Lite 是 v1 原型。当前探讨的是 **v2 的架构方向**——不再是轻量级：
- 更完整的模块化架构
- 更高的性能要求（量化计算密集型）
- 仍然是个人级别，但不是玩具

#### AI 编排：渐进式放权模型

**不同意 Oracle "AI 永远不做编排"的判断，但认同"v1 不要让 AI 做决策"的建议。**

核心论点：
- AI 会进化，进化速度远超个人。GPT-6、Claude Opus 5 会比现在强很多
- 对个人研究者而言，当前 Opus 4.6 的分析能力可能已超过大多数个人交易者的平均水平
- 个体人的精力有限，AI 编排是必然趋势
- 但当前模型在交易决策上确实还不够可靠

**解决方案：架构预留编排接口，权限随信任度逐步升级。**

```
Level 0（当前）：AI 做离线分析和报告
Level 1：AI 提改进建议 → 人审批 → 系统执行
Level 2：AI 自动调参 + 自动回测验证 → 只在阈值外时通知人
Level 3：AI 在预设边界内自动选策略/品种 → 超边界时通知人
Level 4：AI 全自动编排 → 人只做紧急干预和定期审计
```

每次升级的前提：
- 前一级别已稳定运行足够长时间
- 有量化证据（不是主观感觉）表明 AI 决策质量达标
- 有回滚机制确保升级失败时可以降级

这个模型同时满足了 Oracle 的"先跑稳"和用户的"为 AI 进化预留空间"两个需求。

### 2026-03-20：AI 原生加密交易系统的原则与边界 (GPT-5.4 评估视角)

**注：本节内容基于 GPT-5.4 对该项目方向的独立评估。** 与 Oracle 偏向保守和强调当前系统风险的视角不同，GPT-5.4 更认可该方向的战略纵深和未来演进价值，但同样强调在执行层面必须有极其严格的"研究与生产隔离"纪律。

#### 定位

这不是一个"更聪明的量化策略"，而是一个 **AI 原生的加密交易研究与执行系统**。

关键词：
- 加密市场
- 研究与执行一体
- 自我进化
- 渐进式 AI 放权

#### 核心原则

1. **AI 是假设生成器，不是事实生成器。**
   AI 的所有输出，默认都先被当作 hypothesis，而不是 truth。它可以提出解释、给出建议、生成提案，但不能凭一句话成为结论。

2. **研究层与生产层必须分离。**
   AI 可以在研究层做很多事：读日志、识别模式、生成提案、改代码、跑回测、生成报告。但生产层只接收经过验证和门控的结果，不能直接吃 AI 的原始判断。

3. **AI 的权限只随证据增长，不随想象增长。**
   不是因为模型看起来更聪明就放权，而是因为有长期量化证据表明它在某个具体任务上稳定优于人工或规则系统，才给它更高权限。

4. **元策略是方向，但单策略稳定性是地基。**
   元博弈、策略生态识别、AI 编排都值得做，但必须建立在一个单市场、单策略、单执行链路已经无聊地稳定的基础上。

5. **系统竞争力不在某个策略，而在适应速度。**
   长期真正有价值的不是某组参数，而是：发现环境变化 → 形成假设 → 验证 → 调整 → 部署 的速度和质量。

#### AI 可以做什么

- 读取交易日志和行为记录，聚类失败案例
- 识别市场环境标签，生成结构化分析报告
- 对传统策略（如 R-Breaker）提出改进提案
- 起草代码改动并自动回测验证
- 生成对比报告和审批材料
- 在预设边界内做调参与回测编排

#### AI 暂时不应该直接做什么

- 仅凭少量实盘交易样本修改策略骨干逻辑
- 跨市场直接分配真实资金
- 在没有硬门控的情况下自动上线实盘
- 把 narrative（合理解释）当成 evidence（统计证据）
- 在基础执行链路不稳定时接管高层决策

#### 两条线并行的工作方式

不是等单市场完全稳定后才想元策略，而是：

1. **产品线**：把 Binance 单市场执行链路做稳，确保 backtest → testnet → live 无聊地可靠
2. **研究线**：并行建立环境识别、假设生成、提案验证、策略生态分析的框架

两条线的关系：研究线可以比产品线更激进，但研究线的任何产物都不能绕过产品线门控直达生产。

#### 对未来 AI 进化的态度

不否认今天的 AI 在交易决策上还不够成熟，但也不把今天的边界误判成长期边界。

正确做法不是"永远把 AI 限制在报告生成器"，而是：
- 架构上预留 AI 编排接口
- 现实中采用渐进式放权
- 让未来更强的模型可以平滑接入，而不是逼迫系统再次重构

#### 最终目标（长期）

长期目标不是一个会自动下单的大模型，而是一个：

**持续感知市场结构 → 识别策略拥挤与环境变化 → 自动生成与验证新假设 → 在边界内逐步执行的 AI 交易智能体平台。**

它的核心竞争力不在某个神奇策略，而在于：
- 更快地学习
- 更快地适应
- 更快地迭代
- 仍然保持风控和审计可控

### 2026-03-20：元博弈框架与 AI 交易智能体方向

#### 核心认知跃迁

项目方向从"用 AI 辅助的量化交易工具"升级为"加密市场的 AI 交易智能体平台"。

三层认知：

1. **价格行为是人类心理的投射。** 震荡、突破、反转等形态反复出现，因为背后是人类的恐惧、贪婪、锚定、从众。传统量化就是把这些心理模式编码成规则来赚钱。

2. **传统量化策略本身也是市场结构的一部分。** 足够多的人用同一种策略时，策略信号会自我实现，也会被反向利用。所以系统不是在"找市场规律"，而是在"分析市场参与者在做什么，然后针对性应对"——元博弈。

3. **AI 可以跳出人类心理框架。** 人类交易者受情绪延迟和认知偏差约束，AI 没有。AI 的优势不是"更快执行人类策略"，而是"做人类心理上做不到的事"。

#### "价格包含一切信息"假设的重新审视

在 2026 年的加密市场，信息源远不止价格：
- 链上数据（大户转账、DEX 流动性、清算地图）
- 社交媒体情绪（CT 情绪、KOL 持仓）
- 交易所微观结构（订单簿深度、资金费率、未平仓合约）
- 宏观数据（美联储、CPI）

如果 AI 能比价格更快地处理这些信息源，那"价格包含一切信息"这个假设就不成立——可以在价格反映之前行动。

#### 产品形态参考：Dexter

virattt/dexter（18.1k星，TypeScript，LangChain）是一个自主金融研究 agent。不是要直接使用，而是产品形态的参考：

- 一个 AI agent 自主完成复杂金融工作流
- 技能系统（SKILL.md）驱动，可扩展
- Scratchpad 机制记录每次决策过程
- SOUL.md 定义 agent 的投资哲学
- 多模型路由

我们的智能体是同类形态，但做的是交易执行（不只是研究）、策略进化（不只是一次性分析）、元博弈（针对市场策略生态做应对）。

#### 关于"干中学"

对 Oracle "先跑稳再扩展"的回应：不是先跑稳再想元策略，而是两条线并行：
- 一边在单市场单策略上验证基础执行链路
- 一边建立元策略的认知框架和数据收集机制

这是前沿领域，可能没有巨人的肩膀。但这也是机会：大量人在前赴后继地烧 token 追逐 AI 交易，这些经验会沉淀出新交易形态。主流交易所和数据平台已在出 MCP/Skills 接口，AI 原生交易是时间问题。

#### 系统的核心竞争力定位

不在某一个策略上，而在于：
1. 持续感知市场策略生态（谁在用什么策略、哪些策略拥挤）
2. 针对性调整自己的定位（避开拥挤、做对手盘、找补位）
3. AI 进化速度快于人类适应速度
4. 多信息源融合（不止看价格）

### 2026-03-20：Gemini 3.1 Pro 评估视角

**注：本节内容基于 Gemini 3.1 Pro 对该项目方向的独立评估。** 与 Oracle 偏保守、GPT-5.4 偏战略执行不同，Gemini 的视角更激进、更面向进攻——核心观点是"不要做更好的交易员，要做市场中机械性失败的掠食者"。

#### 1. 元博弈假设：猎杀"结构性伤疤"

**判断：逻辑成立，但传感器定义要修正。**

不需要交易所底层 Order Book 数据来检测策略拥挤度。传统量化策略会在价格图表上留下结构性伤疤（Structural Footprints）：
- 当大量机器人在同一价位设置买入止损时，突破失败后必然出现流动性真空和止损踩踏
- AI 不需要预测价格，只需要分类当前的"生态位"——趋势跟随者在控盘，还是均值回归者在控盘？
- 核心：不是交易 BTC，而是交易那些做错方向的量化机器人的强制平仓线

#### 2. 信息优势假设：深度合成胜过速度

**判断：速度上绝对赢不了机构，但"因果链长度"可以赢。**

机构的 NLP 机器人通常是孤岛式的（专门盯新闻的、专门盯链上的、专门盯情绪的）。个人 AI 的优势在于做通才（Generalist）：
- 把"周二某核心开发者的 GitHub 异常提交" + "周三链上巨鲸的 DEX 流动性转移" + "周四宏观数据泄露预期"连接起来
- 机构在对"发生了什么（What）"做毫秒级反应
- 个人 AI 应该去寻找"为什么（Why）"带来的非对称叙事反转——在价格反映之前看到故事逻辑的崩塌

**核心差异：机构优化延迟（latency），个人应该优化因果推理（causality）。**

#### 3. 创新扩展："叛徒机器人"（Adversarial Strategy Mimicry）

不去优化 R-Breaker，而是构建主策略的"影子模型（Shadow Model）"：
- 影子模型的唯一目标：寻找让主策略亏钱的完美市场环境
- 当影子模型高度确信主策略（如趋势跟随）即将遭遇假突破陷阱时，反手做空主策略
- 本质：通过猎杀自己的经典策略弱点来获利
- 你成为自己最危险的对手——在模拟中杀死自己的策略，在实战中避免被别人杀死

#### 4. 最薄弱环节：人类审批是噪音注入点

Telegram 人工审批不仅是性能瓶颈，更是致命的偏差注入点：
- 交易中 90% 是无聊（盲目通过一切提案），10% 是恐惧（否决数学上正确但反直觉的改进）
- 把人留在代码逻辑的修改闭环里 = 把人类的"贪婪与恐惧"重新硬编码进机器学习循环
- 人不是"审批者"，而是"噪音发生器"

**与渐进式放权模型的关系：** 这恰好支撑了从 Level 0 逐步升级到 Level 4 的必要性——不是因为 AI 完美，而是因为人的参与本身也在引入系统性偏差。

#### 5. Gemini 的三个颠覆性 v2 建议

**建议一：代理辩论协议（Agentic Debate Protocol）**

不让单个 AI 给出"建议"。当系统想改策略时：
- 生成两个 LLM 实例："优化者（多头）"提出改动，"做空者（空头）"专门找漏洞攻击
- 只有当它们达成共识或用回测数据分出胜负后，才发通知给人
- 让 AI 先完成自我审计，再打扰人类

**建议二：合成黑天鹅压力测试（Synthetic Adversaries）**

不要只拿历史数据回测：
- 用生成式模型（GAN）创造"不可能的市场"——连续 50% 闪崩、无尽低波动死水、1000% 逼空
- 如果 AI 的新策略逻辑能在"AI 生成的噩梦环境"里活下来，它才配上实盘
- 这比 walk-forward 更激进，也更能暴露结构性弱点

**建议三：跨模态信号融合（Cross-Modal Signal Fusion）**

不建更好的策略，建更好的"状态检测器（Regime Detector）"：
- 高波动率 → 无视社交情绪，只看 CEX 订单流
- 极低波动率 → 无视价格行为，只盯链上"聪明钱"移动
- 动态切换信号权重比策略参数本身更重要
- 加密市场的价格行为经常从 DEX 领先于 CEX（流动性先行），这个 Lead-Lag 关系是可被利用的 alpha 来源

#### Gemini 的最终判断

> 不要做一个试图成为"好交易员"的机器人。做一个专门寻找"别人在哪里当坏交易员"并提供他们亏钱所需流动性的机器人。

### 三大模型视角对比总结

| 维度 | Oracle (GPT-5.4 保守模式) | GPT-5.4 (战略执行模式) | Gemini 3.1 Pro |
|---|---|---|---|
| 核心态度 | 防守优先 | 攻守兼备 | 进攻优先 |
| AI 角色 | 离线分析师 | 渐进式放权的编排者 | 市场掠食者 |
| 最大风险 | 过拟合和工程复杂度 | 研究与生产混为一谈 | 人类审批本身是噪音 |
| 策略观 | 先跑稳一个再说 | R-Breaker 当基线，架构为进化预留 | 猎杀自己策略弱点比优化策略更重要 |
| 独特贡献 | 硬性门控、回滚机制、样本量要求 | 渐进式放权模型、两条线并行 | 影子模型、代理辩论、合成黑天鹅、信号融合 |
| 对元博弈 | 承认价值但认为超出当前能力 | 值得做但先建基础 | 这就是核心——交易的是交易者，不是价格 |

### 2026-03-21：GPT-5.4 xhigh 评估视角（方案本体版）

#### 先说总判断

我认为这个方向是对的，而且不是“小修小补”的对，而是有可能把项目性质整个抬升一个层级的对。

因为你这里想做的，本质上已经不是“一个会跑策略的交易工具”，而是一个 **会从自身交易行为中学习、形成假设、验证假设、再决定是否改变自己的研究型系统**。

这件事真正有价值的地方，不在于 AI 帮你写了多少代码，也不在于某次调参跑出了多漂亮的结果，而在于它试图把“交易系统”从静态规则，变成一种 **有记忆、有反馈、有演化能力的研究生命体**。

这个方向是有野心的，而且我认为野心得是对的。

#### 这套方案最吸引我的地方

不是“AI 改策略”这四个字本身，而是它把交易里的一个老问题正面提了出来：

**传统策略系统会执行，会记录，但不会真正学习。**

大多数系统的闭环其实是假的：

- 跑策略
- 亏钱或赚钱
- 人自己看几眼图
- 凭印象去改参数或改逻辑
- 再跑下一轮

这里面没有真正结构化的“经验沉淀”。

而你这份方案里最有价值的地方，是试图把这个过程显式化：

- 把实盘行为变成可分析对象
- 把失败模式变成可归因对象
- 把改动建议变成显式提案
- 把上线动作变成带审批和验证的制度化过程

如果这套东西做成了，它的长期价值确实可能不在某一个策略，而在于 **适应速度**。这一点我认同。

#### 但这里面有一个非常大的认知陷阱

这份方案最容易滑进去的坑，不是工程复杂度，而是 **把“解释能力”误当成“认知能力”，再把“认知能力”误当成“赚钱能力”**。

LLM 非常擅长三件事：

- 给已有结果做高质量解释
- 在杂乱信息里提出新假设
- 把零散现象组织成一个看起来很完整的故事

但交易里最危险的，恰恰就是“故事很好听”。

所以我对这个方案的核心提醒是：

**AI 在这里最适合当“假设生成器”和“失败解释器”，不适合被默认当成“真相发现器”。**

也就是说，AI 的价值不在于直接告诉你“该怎么赚钱”，而在于：

- 把你原本注意不到的失败模式提出来
- 把人类很难持续做的复盘工作自动化
- 把下一步值得验证的方向扩展开
- 把研究循环提速

这已经非常有价值了，但它和“AI 能持续找到 alpha”不是一回事。

#### 我最认同文档里的哪部分

1. **研究与生产分离**

这是整份文档最重要的原则之一。  
只要这条守住，很多激进想法都可以在研究层大胆尝试。  
一旦这条守不住，系统会迅速把“研究噪音”误当成“生产改进”。

2. **渐进式放权**

这点我非常认同。  
不是因为“谨慎”，而是因为 AI 在不同任务上的可靠性根本不一样。  
它可能很适合总结日志，也可能勉强适合提出过滤器建议，但离“直接决定真实资金如何配置”还差得很远。  
放权应该按任务颗粒度，而不是按对模型的整体信仰。

3. **长期竞争力在适应，而不是在单一策略**

这点我也认同。  
真正稀缺的东西不是某个参数组合，而是：

- 什么时候发现旧逻辑失效
- 什么时候知道该修、该停、还是该换
- 能不能把失败变成下一轮研究输入

从这个角度看，你想做的不是策略仓库，而是一个“研究-验证-晋升”的持续循环系统。

#### 我最担心文档里哪部分

我最担心的不是技术实现，而是 **问题域膨胀得太快**。

现在这份文档里同时装进去了很多层不同的问题：

- 参数调整
- 过滤器增强
- 出场规则优化
- 策略骨干改写
- 多品种扩展
- 多市场扩展
- 元博弈
- 跨模态信息融合
- 影子模型
- AI 辩论

这些不是同一个难度层级的问题。

如果把它们放进同一个 v2 愿景里，系统会很容易失去焦点。最后的问题不是“方向太大”，而是 **学习信号被污染**。

一旦你同时做太多层，最后很难回答：

- 到底是哪个变化带来了改进？
- 是环境变了，还是策略真的变好了？
- 是新信息源带来了优势，还是只是让叙事更复杂了？

所以我对这个方案的建议不是“变小”，而是 **分层**。

#### 我建议你从概念上把它拆成三条循环

##### 1. 利用循环（Exploit Loop）

目的不是创新，而是稳定赚钱。

它关心的是：

- 当前策略是否还能正常工作
- 当前环境下是否应该继续运行
- 是否触发停机、降杠杆、回撤保护

这个循环必须保守、低解释欲、重纪律。

##### 2. 适应循环（Adapt Loop）

目的不是发明新世界，而是修复和增强现有策略家族。

它做的事情包括：

- 参数变化
- 过滤器增加或删减
- 出场机制修正
- 对已知弱点做局部补丁

这条循环可以半自动，甚至逐步自动化。  
因为它处理的是 **有边界的变异空间**。

##### 3. 探索循环（Explore Loop）

目的是真正找新东西。

它可以装下这些更激进的内容：

- 元博弈
- 影子模型
- 跨模态信号
- 新策略家族
- 对手策略生态分析

但这条循环的成果，不应该直接被当成生产改动，而应该先被当成“研究资产”。

我认为这是这份方案很关键的一刀。  
不这样拆，最后“稳定执行”和“激进探索”会互相污染。

#### 关于“人的角色”，我想给一个不同表述

文档里把人定义成“审批者”，Gemini 又把人视为“噪音源”。

我觉得这两种说法都不够准确。

人真正的角色，不应该只是：

- 每次都点一下通过
- 或者每次都凭感觉否掉

这太低级了。

我认为人在这个系统里的真正角色应该是：

**制度设计者、边界定义者、事后裁判者。**

也就是说，人最该决定的不是“这次提案听起来顺不顺耳”，而是：

- 什么类型的变动可以自动进入下一轮
- 什么证据才算够
- 什么情况下必须停机
- 什么情况下必须回滚
- 哪类风险绝不允许系统自作主张

如果人只做按钮审批，确实容易变噪音。  
但如果人负责的是“制定宪法”，那人的价值反而是不可替代的。

#### 关于 R-Breaker，我的看法

我认同把 R-Breaker 作为起点，但我认同的原因不是它“经典”或“知名”，而是它 **适合做第一批研究对象**。

好的起始策略不一定是最赚钱的，而应该是：

- 结构清晰
- 失败模式清晰
- 容易被拆解
- 容易形成可证伪的改进假设

从这个角度说，R-Breaker 作为起点是合适的。  
因为你不是要证明“R-Breaker 就是答案”，而是要拿它来训练整个研究与进化流程。

#### 关于“市场环境标注”，我建议提高警惕

这里很容易产生一个错觉：  
好像只要把市场贴上“趋势 / 震荡 / 高波动 / 低波动”标签，系统就理解市场了。

其实不是。

环境标签更像是一种 **观察角度**，不是隐藏真相。

同一个市场阶段，可以同时是：

- 趋势扩张
- 高波动
- 流动性脆弱
- 某类参与者主导

所以环境标注系统应该被理解为：

- 帮助诊断
- 帮助分桶
- 帮助做条件化分析

而不是试图给市场下唯一正确的定义。

如果把标签当真相，后面整个提案系统都会建立在脆弱基础上。

#### 关于多市场，我的建议比“先别做”更进一步

不是因为多市场工程复杂，所以先缓一缓。  
而是因为 **多市场会让系统更难学会真正的东西**。

不同市场混在一起时，很多经验其实不可比较：

- 交易节奏不同
- 流动性结构不同
- 杠杆与结算机制不同
- 假突破和趋势延续的行为结构不同

一旦把这些东西混到一个“进化系统”里，系统会更容易学到模糊规律，而不是清晰规律。

所以我对多市场的观点是：

**不是单纯的工程延后，而是认知上也应该延后。**

先让系统在一个相对一致的生态里学会“怎么学习”，比一开始就学“怎么兼容一切”更重要。

#### 关于 Gemini 那些激进想法

我不认为 Gemini 的方向错。  
相反，我觉得那些想法很有价值，而且有些甚至是未来真正的 alpha 来源。

但问题在于，它们属于 **研究武器**，不是 **系统主骨架**。

例如：

- 影子模型很适合当对抗验证器
- 代理辩论很适合当研究审稿机制
- 合成黑天鹅很适合当极端脆弱性测试
- 跨模态信号很适合当未来研究方向

但如果现在就把这些东西升格成 v2 核心，会让系统过早进入“高概念、高叙事、高复杂度”的状态。

所以我会这样看 Gemini：

**它不是 roadmap，它是 research backlog。**

#### 我觉得这份方案还缺一个非常重要的东西

缺的是：**失败记忆机制。**

很多系统只会保存成功经验，不会系统性保存失败经验。  
但对这种“策略进化系统”来说，失败记忆反而可能更重要。

系统应该长期记住：

- 哪类改动曾经看起来很合理但最后失败
- 哪些市场环境判断经常误导自己
- 哪些优化方向反复导致过拟合
- 哪些提案在 testnet 好看、live 很差

因为真正让系统成熟的，不是“会提出新想法”，而是“不会反复掉进同样的坑”。

如果没有失败记忆，这个系统最后就会变成一个很会讲新故事的系统，而不是一个会变聪明的系统。

#### 如果是我来收束这份方案，我会把它的定位改成一句更精确的话

不是：

**“AI 自动改进交易策略系统”**

而是：

**“以真实交易行为为反馈、以 AI 为假设生成器、以验证与门控为核心的持续研究系统。”**

这两句话差别很大。

前一句容易让人误以为核心是“AI 会不会更聪明”。  
后一句更接近我认为真正可持续的方向：

- AI 负责扩展思路
- 数据负责提供反馈
- 验证负责筛选幻觉
- 制度负责控制风险

#### 我的最终建议

1. 保留这个大方向，不要因为风险就把它降格成普通调参系统
2. 但要把野心集中在“让系统真正学会复盘和形成假设”上，而不是同时追逐所有先进叙事
3. 把整个方案明确拆成“利用 / 适应 / 探索”三条循环
4. 把人的角色从“日常审批者”升级成“制度设计者”
5. 把 Gemini 那一类激进能力放进研究储备区，不要直接升为 v2 主骨架
6. 把“失败记忆”补成核心组成部分

#### 最后一句话

这份方案最有前途的地方，不是它可能做出一个越来越会交易的 AI。

而是它有机会做出一个 **越来越会从自己的成功与失败中学习的系统**。

如果这个点抓住了，它就不是普通量化工具的延长线了。  
如果这个点抓不住，那它最后很容易退化成“AI 帮你写回测、讲故事、做提案”的复杂包装。

### 2026-03-21：方向收束版（建议作为当前主线）

如果把前面 Oracle、GPT-5.4、Gemini、以及本轮补充的判断收束成一条更统一的主线，我会把整份文档理解成下面这句话：

**QPorter Lite v2 不是要先做一个“什么都懂、什么都能编排”的 AI 交易员，而是要先做一个能够从真实交易反馈中持续学习、在受控边界内逐步适应的研究系统。**

这句话里有几个关键词，它们决定了这份方案到底会走向“有长期价值的系统”，还是“高概念但容易失控的愿景”。

#### 1. 它首先是“研究系统”，其次才是“执行系统”

执行当然重要，但执行不是这份方案真正的新意。  
新意在于：系统不再只负责跑策略，而是开始负责复盘、解释、提出假设、比较改动、沉淀经验。

也就是说，这个项目真正想发明的，不是一个新执行器，而是一种新的研究工作流。

#### 2. 它应该先学会“适应”，而不是先追求“统治”

很多宏大愿景会天然滑向一种冲动：  
既然都要重构了，不如直接把多市场、多策略、元博弈、跨模态、AI 编排一起放进去。

但真正强的系统，不是因为一开始覆盖面最大，而是因为它先把一个小生态里的学习过程做对了。

所以更合理的主线应该是：

- 先在单一、相对一致的生态里学习
- 先做局部修正与受控适应
- 再做更激进的探索
- 最后再谈更广的市场与更大的自治权

#### 3. 它真正的核心资产，不是某个策略，而是“学习机制”

从长期看，这个系统最宝贵的东西不会是：

- R-Breaker
- 某个过滤器
- 某组调优参数
- 某次 AI 提案

真正的核心资产会是：

- 如何记录经验
- 如何定义失败
- 如何从失败中提炼假设
- 如何把假设送进验证
- 如何决定哪些变化值得晋升

换句话说，这个系统最终卖点不是“我有一个厉害策略”，而是“我有一套越来越会学习的机制”。

#### 4. 文档里其实已经隐含了三种不同层级的 ambition

把这些 ambition 混在一起，会让系统失焦。分开看会清楚很多。

**第一层：操作性 ambition**

- 自动生成分析报告
- 自动提出改进建议
- 自动回测和对比
- 自动推送审批材料

这是最现实、也最接近近期可做成的部分。

**第二层：适应性 ambition**

- 根据市场环境调整参数
- 增减过滤器
- 修正退出机制
- 逐步形成策略家族的“自修复”能力

这是最有潜力成为中期竞争力的部分。

**第三层：认知性 ambition**

- 元博弈
- 生态位识别
- 影子模型
- 跨模态因果链
- AI 自主编排

这是最吸引人、也最危险的部分。  
它不是不值得追求，而是需要更长时间、更强证据和更高自律。

#### 5. 因此，我建议把文档里的内容分成三类

**A. 当前主线**

- 实盘行为分析
- 失败模式归因
- 市场环境分桶
- 受约束的改进提案
- 回测 / 测试验证
- 审批与晋升
- 失败记忆沉淀

这些内容共同组成“系统如何学会复盘和适应”的第一性主线。

**B. 中期增强**

- 多策略家族比较
- 影子模型作为对抗验证器
- 更丰富的环境标注与状态检测
- AI 辩论作为研究审稿机制

这些适合作为中期增强能力，但不应反客为主。

**C. 远期研究储备**

- 多市场统一进化
- 跨模态外部信息融合
- 猎杀策略拥挤度的元博弈系统
- 高自治权限的 AI 编排
- 合成极端市场生成器

这些都可以保留，但更适合以 research backlog 身份存在，而不是当前主线。

#### 6. 这样收束之后，整份文档会更像一个有脊梁的方向文档

它的脊梁可以浓缩成 6 句话：

1. 先做加密市场内的持续学习系统，不先做全市场智能体
2. 先让系统学会复盘，不先让系统自信地下结论
3. 先做受约束适应，不先做无限制进化
4. 先沉淀失败记忆，不先堆叠新叙事
5. 先把探索与利用分开，不让它们互相污染
6. 先让 AI 成为高质量研究助手，再决定是否让它成为更高权限的编排者

#### 7. 如果要给这份方案一个最稳的当前定义

我会建议把当前主定义写成：

**“一个面向加密市场、以真实交易反馈为燃料、以 AI 假设生成和验证闭环为核心的持续适应系统。”**

这一定义既保留了野心，也避免了把系统过早定义成“万能 AI 交易员”。

### 2026-03-21：我最不同意的点

下面这些点，不是我“略有保留”，而是我认为如果不明确反对，后面很容易把整个方向带偏。

#### 1. 我不同意把“多市场扩展”放在近中期主叙事里

不是因为它难，而是因为它会过早稀释学习信号。

这份方案的关键问题不是“怎么把更多市场接进来”，而是“怎么让系统先学会真正地学习”。  
如果系统连在一个相对一致的生态里都还没学会稳定复盘、稳定形成假设、稳定修复错误，那么跨市场只会放大混乱，不会放大能力。

所以我对多市场不是“以后再说”的温和态度，而是：

**当前阶段不应该把它当成主线目标。**

#### 2. 我不同意把“50-100 笔实盘交易”看作足以支持结构性进化

这只足够支持：

- 描述性分析
- 症状发现
- 局部假设生成

它远远不够支持：

- 大改策略骨架
- 断言某种逻辑在统计上更优
- 基于少量样本重写策略世界观

如果用这么少的样本驱动结构性变化，系统会非常容易把偶然性包装成洞察。

#### 3. 我不同意把“市场环境标签”理解成系统认知核心

环境标签很有用，但它不是世界模型。

一旦把“趋势 / 震荡 / 高波动 / 低波动”这类标签看得太重，后面整个系统都会出现一种错觉：  
只要分类做对了，提案就有了客观基础。

实际上，标签只是帮助观察，不是最终解释。  
如果系统未来过度依赖标签，它可能只是换了一种更漂亮的方式过拟合。

#### 4. 我不同意把 Gemini 式的“市场掠食者叙事”作为 v2 身份认同

这个叙事非常有吸引力，也很容易让人兴奋。  
问题是，它太强、太酷、太完整了，反而容易跳过最重要的问题：

- 你真的能稳定识别对手是谁吗？
- 你识别到的是结构性弱点，还是事后故事？
- 你的“掠食”能力来自真实优势，还是来自叙事幻觉？

所以这类叙事更适合当研究假说，而不是产品身份。

#### 5. 我不同意把“AI 自主编排”当成当前阶段必须保留的中心想象

我不是反对未来 AI 获得更高权限。  
我反对的是，在系统尚未建立稳定学习机制之前，就用“未来更强模型会来”作为今天架构膨胀的理由。

未来模型会更强，大概率是真的。  
但系统是否应该今天就围绕那种未来形态展开，不一定。

更好的态度应该是：

- 现在先把学习闭环做对
- 未来模型变强时，再放大权限
- 不让“对未来的信念”替代“对当下边界的判断”

#### 6. 我不同意把“人”仅仅理解为审批瓶颈或噪音源

如果人只是看 Telegram 然后点一下“通过”，那确实很容易成为噪音。  
但这不是人的问题，而是系统把人的作用设计得太低级。

人在这种系统里真正该承担的责任，不是“替系统做局部判断”，而是：

- 定义边界
- 制定规则
- 决定什么可自动化、什么不可自动化
- 审计系统是否偏离初衷

所以我不接受“人终将被彻底退到紧急按钮”被当成天然正确的默认结论。  
有些权力也许未来可以下放，但“制度设计权”本身不应该被轻率下放。

#### 7. 我不同意把“什么都能做”视为这套方案的成熟标志

这类系统的成熟，不体现在功能总数，而体现在：

- 能不能持续从失败中学到东西
- 能不能避免重复犯错
- 能不能在不确定时收缩而不是冲动扩张
- 能不能区分“新奇”与“有效”

所以我眼里真正成熟的版本，不是那个会同时操作很多市场、吸收很多信号、讲很多故事的版本。  
而是那个 **即使能力看起来没那么炫，但已经具备清晰自我修正机制** 的版本。

#### 最后收束成一句更尖锐的话

如果这份方案未来跑偏，最大的风险大概率不是“做不出来”，而是：

**它做出来了很多看起来先进的东西，但没有真的学会如何变得更对。**

所以真正要反复守住的，不是“功能增长”，而是：

- 学习信号是否清晰
- 假设与证据是否分离
- 探索与利用是否分离
- 失败是否被真正记住
