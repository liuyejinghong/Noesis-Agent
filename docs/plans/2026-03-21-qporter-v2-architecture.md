## 一、项目定位

**一句话定义：**

> 一个面向加密市场、以真实交易反馈为燃料、以 AI 假设生成和验证闭环为核心的持续适应系统。

**不是什么：**
- 不是一个"更好的量化策略"
- 不是一个"AI 自动赚钱机器"
- 不是一个全球多市场交易平台

**是什么：**
- 一个 AI 原生的加密交易研究与执行系统
- 一个能从自身交易行为中持续学习的研究生命体
- 一个渐进式 AI 放权的智能体平台

---

## 二、核心原则（宪法级）

1. **AI 是假设生成器，不是事实生成器**
2. **研究层与生产层必须分离**
3. **AI 权限只随证据增长，不随想象增长**
4. **先学会适应，再追求统治**
5. **系统竞争力在适应速度，不在某个策略**
6. **失败记忆和成功记忆同样重要**
7. **探索与利用分开，不让它们互相污染**

| 失败记忆 | 所有失败案例、误判、过拟合教训的结构化存档 |
| 审批队列 | 等待审批的提案，带完整证据链和回测对比 |
| 策略仓库 | 策略家族、版本管理、跨版本表现对比 |
| 交易监控 | 实时持仓、PnL、风控状态（v1 已有，微调） |
| Agent 设置 | 权限级别、模型配置、心跳频率、边界规则 |
| 系统设置 | 交易所连接、通知渠道（v1 已有，微调） |

**Telegram（辅助）：**
- 紧急告警推送
- 移动端快速审批（通过/拒绝）
- Agent 状态摘要推送

### 4.2 Agent 智能体层

#### Agent 类型（近期）

| Agent | 输入 | 输出 | 触发方式 |
|---|---|---|---|
| **分析 Agent** | 交易行为记录 + 市场数据 | 结构化分析报告 | 月级心跳 / 手动 |
| **提案 Agent** | 分析报告 + 当前策略源码 | 改进提案（含代码变更） | 分析报告完成后 |
| **验证 Agent** | 提案 + 历史数据 | 回测对比报告 | 提案生成后 |

#### Agent 类型（中期）

| Agent | 输入 | 输出 | 触发方式 |
|---|---|---|---|
| **辩论 Agent** | 提案 | 多空双方论证 + 共识/分歧 | 验证通过后 |
| **环境 Agent** | 市场数据流 | 市场状态标签（趋势/震荡/高波动） | 持续运行 |

#### Agent 类型（远期研究储备）

| Agent | 输入 | 输出 |
|---|---|---|
| **影子 Agent** | 主策略 + 市场数据 | 主策略弱点报告 |
| **编排 Agent** | 所有 Agent 输出 + 市场状态 | 策略/品种/参数选择建议 |
| **生态 Agent** | 价格行为 + 链上数据 | 策略拥挤度估计 |

#### Agent 基础设施

**记忆系统：**

三类记忆，统一存储在 SQLite，查询通过业务方法接口暴露（不暴露裸 SQL），未来可在不改调用方的前提下加向量搜索层。

| 类型 | 用途 | 生命周期 | 存储 |
|---|---|---|---|
| **短期记忆（Scratchpad）** | 单次任务的工具调用、中间推理、结果 | 任务结束后归档 | SQLite 行 / JSONL |
| **长期记忆（Knowledge Base）** | 分析报告、提案历史、回测对比、环境标注、策略版本 | 永久 | SQLite + FTS5 |
| **失败记忆（Failure Log）** | 被否决的提案、实盘不及回测、过拟合案例、误判、系统错误 | 永久，提案前强制查询 | SQLite + FTS5 |

查询接口设计原则：
```
memory.query_failures(strategy_family, change_type, tags) → list[FailureRecord]
memory.search_similar(description, top_k) → list[MemoryRecord]
memory.get_proposals(strategy_id, status) → list[ProposalRecord]
memory.get_reports(period) → list[AnalysisReport]
```

`search_similar` 当前用 FTS5 关键词搜索实现，未来可替换为向量搜索，调用方不变。
所有原始文本始终存在 SQLite 主表，向量索引随时可从原始数据补建，不存在数据丢失风险。

**审批协议：**

提案从生成到上线实盘，经过 5 道自动门控 + 2 次人工审批。能自动拒绝的都自动拒绝，不打扰人。

提案状态机：
```
draft → gate_1_memory → gate_2_backtest → gate_3_walkforward
  → pending_approval → approved → testnet_deployed
  → gate_4_min_period → gate_5_testnet_performance
  → pending_live_approval → live_deployed → monitoring
  → [auto_rollback] / [graduated]
```

自动门控定义：

| 门控 | 检查内容 | 拒绝条件 | 通过条件 |
|---|---|---|---|
| Gate 1: 失败记忆 | 查询失败记忆库 | 命中同策略+同类型+同原因的历史失败 | 无命中或前提明确不同 |
| Gate 2: 回测验证 | 新版本 vs 旧版本 | 夏普下降 OR 回撤增大 OR 胜率显著下降 | 关键指标不劣于旧版本 |
| Gate 3: Walk-forward | 样本外验证 | 样本外衰减超过阈值 | 衰减在可接受范围内 |
| Gate 4: 测试网运行期 | 运行天数 + 交易笔数 | 未满最低要求 | 达标 |
| Gate 5: 测试网表现 | 实际 vs 回测预期 | 显著偏离 | 偏离可接受 |

人工审批节点：
1. **提案审批**（Gate 1-3 通过后）：Web 审批队列展示完整证据链。Telegram 推送摘要，可快速通过/拒绝。
2. **实盘晋升审批**（Gate 4-5 通过后）：测试网验证报告。人确认后自动切换实盘。

实盘自动回滚触发条件（任一触发 → 自动回滚到上一版本 + 通知）：
- 日亏损超过熔断值
- 连续 5 笔亏损
- 实际表现低于回测预期的 -50%
- 紧急停机信号

回滚操作：停止当前版本 → 切换上一版本 → 记录失败到失败记忆 → 通知人。

门控初始阈值（可随经验调整，调整本身记录到长期记忆）：

| 参数 | 初始值 |
|---|---|
| Walk-forward 最大衰减率 | 30% |
| 测试网最低运行天数 | 14 天 |
| 测试网最低交易笔数 | 20 笔 |
| 实盘偏离回滚阈值 | 回测预期的 -50% |
| 连续亏损回滚触发 | 5 笔 |

**技能注册表：**
把已有引擎的能力封装为 Agent 可调用的技能：
- `run_backtest(strategy_id, symbol, timeframe, params)` → 回测引擎
- `optimize_params(strategy_id, search_space)` → 优化引擎
- `get_trade_records(period)` → 交易记录查询
- `get_market_data(symbol, timeframe, range)` → 数据层
- `compute_indicators(data, indicators)` → TA-Lib
- `classify_regime(data)` → 环境识别
- `submit_proposal(proposal)` → 审批队列

**多模型路由：**

通过 LiteLLM 统一接入，支持官方 API 和第三方中转站（自定义 api_base）。

**不按模型能力分级，按业务角色定义。** 每个 Agent 是一个完整的角色定义 = system prompt + 模型选择 + fallback + 可用工具 + 输出格式，全部可在 TOML 配置中调整，不改代码。

Agent 角色与模型分配：

| Agent 角色 | 职责 | 默认模型 | Fallback | 用 LLM？ |
|---|---|---|---|---|
| 策略分析师 | 读交易记录 + 市场数据 → 分析报告 | Opus | GPT-5.4 | 是 |
| 策略改进师 | 读报告 + 策略源码 → 改进提案 | GPT-5.4 | Opus | 是 |
| 提案验证员 | 回测对比 + walk-forward 验证 | Sonnet | Flash | 是 |
| 多头辩手 | 论证提案合理性 | Opus | GPT-5.4 | 是 |
| 空头辩手 | 攻击提案弱点 | GPT-5.4 | Opus | 是 |
| 报告撰写员 | 整理为可读中文报告 | Flash | Sonnet | 是 |
| 环境观察员 | 市场状态分类 | 规则引擎 | — | 否 |

设计要点：
- 多头/空头辩手刻意用不同模型，避免同模型系统性偏差
- 环境观察员不用 LLM（TA-Lib + HMM 规则引擎）
- Fallback 是角色级的，不是全局的
- 所有角色配置（模型/prompt/工具/输出格式）都在 TOML 里，换模型不碰代码

路由策略：
- 能不用 LLM 的地方就不用（心跳/风控/门控/环境分类用规则引擎）
- 日常任务走便宜渠道（中转站/Flash 模型）
- 关键任务走高质量模型（Opus/GPT-5.4）
- Fallback 按角色能力需求切换同级模型，不是按渠道

支持的接入方式：
- 官方 API（各厂商独立 API key）
- 中转站（自定义 api_base，兼容 OpenAI 格式）
- 本地模型（Ollama）
- 注意：ChatGPT Plus 等订阅制会员不能直接用于 API 调用，系统需要独立的 API 接入

| 任务类型 | 推荐模型 | 频率 | 成本级别 |
|---|---|---|---|
| 心跳/健康检查/风控 | 不用 LLM | 分钟级 | 零 |
| 环境分类 | Sonnet / Flash | 每日 | 低 |
| 月度分析 | Opus / GPT-5.4 | 每月 | 中 |
| 提案生成 | Opus / GPT-5.4 | 偶尔 | 高 |
| 报告生成 | Flash | 偶尔 | 低 |
| 辩论协议 | 两个不同模型 | 偶尔 | 中 |

配置方式：TOML 里声明 provider（含 api_base/api_key），路由规则按任务类型映射。
敏感值（API key）走环境变量，不入配置文件。

**调度器：**
- 分钟级心跳：健康检查、风控状态同步、异常自愈
- 日级心跳：环境状态更新、交易记录累积检查
- 月级心跳：触发分析 Agent → 提案 Agent → 验证闭环
- 事件驱动：紧急停机、异常回撤、策略失效信号

## 五、三条循环

### 利用循环（Exploit Loop）

**目的：稳定执行，不创新。**

- 当前策略是否正常工作
- 当前环境是否应该继续运行
- 是否触发停机、降杠杆、回撤保护

保守、低解释欲、重纪律。规则引擎驱动，不用 LLM。

### 适应循环（Adapt Loop）

**目的：修复和增强现有策略家族。**

- 参数变化
- 过滤器增减
- 出场机制修正
- 对已知弱点做局部补丁

可半自动，逐步自动化。处理的是有边界的变异空间。

### 探索循环（Explore Loop）

**目的：找新东西。**

- 元博弈
- 影子模型
- 跨模态信号
- 新策略家族
- 对手策略生态分析

成果不直接进生产，先当"研究资产"。

## 六、渐进式 AI 放权模型

```
Level 0（v2 初期）：AI 做离线分析和报告
Level 1：AI 提改进建议 → 人审批 → 系统执行
Level 2：AI 自动调参 + 自动回测验证 → 只在阈值外时通知人
Level 3：AI 在预设边界内自动选策略/品种 → 超边界时通知人
Level 4：AI 全自动编排 → 人只做紧急干预和定期审计
```

升级条件：
- 前一级别已稳定运行足够长时间
- 有量化证据表明 AI 决策质量达标
- 有回滚机制确保升级失败时可以降级

## 附录：关键设计决策记录

| 决策 | 结论 | 理由 |
|---|---|---|
| 推倒重写 vs 渐进演进 | **渐进演进** | 核心抽象（StrategyBase + ExecutionAdapter）已是插件式，回测/优化/日志都通用可复用 |
| 市场范围 | **只做加密货币** | 准入最低、接口统一、CEX→DEX 扩展最简单 |
| AI 角色 | **渐进式放权** | 初期做研究助手，随证据积累逐步升级权限 |
| 交互方式 | **Web 为主，Telegram 辅助** | Web 是 Agent 控制台，Telegram 做紧急通知和移动审批 |
| 策略起点 | **R-Breaker** | 不是因为它一定赚钱，而是它结构清晰、失败模式已知、适合训练研究流程 |
| Web 前端 | **复用 v1，升级为 Agent 控制台** | ~60% 代码可直接复用，不需要重写框架 |
| Agent 框架 | **PydanticAI + 自建编排层** | 类型安全、结构化输出、最小抽象、类 FastAPI 体验；编排/记忆/调度自建以保持完全可控 |
| 数据层 | **MarketDataAdapter Protocol + 交易所独立实现** | 和 ExecutionAdapter 同样的插件模式；已有通用代码（CSV 读写/catalog/resample）不改，只重写拉取层 |
